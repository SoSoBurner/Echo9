## Phase 3 — Soak — **FAILED (infrastructure, not app)**

**Command:** `SOAK_ITERATIONS=500 npx playwright test src/tests/e2e/soakTest.spec.ts --project chromium`
**Result:** 1 failed. `page.goto('/') → net::ERR_CONNECTION_REFUSED at http://localhost:5173/` mid-iteration.

**Root cause:** Vite dev server (Playwright `webServer`) died mid-soak. The failing call is `page.goto`, not a heap-budget assertion. The 100-iteration soak inside Phase 1 completed with **heap growth 0.00 MB from 15.35 MB baseline** and `save p95 0.70 ms` — the app itself is not the failure surface.

**Why this happened:** the soak runs ~500 boot+choice cycles at ~7-8 s each = ~65-70 min. Vite's dev server accumulates client watchers, HMR sockets, and cached bundles across sessions; long-lived instances hit socket / handle exhaustion. This is a known ceiling for Vite-served long-soak Playwright runs and is called out in `game/CLAUDE.md` (SOAK_ITERATIONS budgeting note, measured 2026-07-07).

**Two mitigations for the real RC:**

1. **Serve the built `dist/` via `npm run play` or `vite preview`** instead of the dev server. Preview is a static file server — no HMR, no watchers, no memory growth from tooling. Update `playwright.config.ts` `webServer.command` to point at `vite preview` for `SOAK_ITERATIONS >= 500`.
2. **Bucket the soak** — five sequential 100-iter runs, harness restart between buckets. Total wall time similar; each Vite process stays inside its ceiling.

Neither mitigation is on the critical path for the pre-RC audit itself — both are `docs/ship-gate.md:25` improvements. But this dry-run cannot claim Phase 3 PASS without them.

Verdict: **FAILED on harness, unproven on budget.** The app's soak behavior in the successful 100-iter run is well within budget; the 500-iter budget hasn't actually been measured because the harness didn't survive the run. Blocker for RC; not a blocker for continued development.

## Phase 5 — Ship-gate signal check — **HONEST NO**

> Does this build convincingly earn ≥10 unsolicited comments mentioning the **trace ledger** or **character silence**?

**No.**

The trace ledger is present (LogDock + AlertCrawler + result cards form a coherent WHY-NOW spine — a genuinely novel surface). But it currently narrates a single directive loop. Ten unsolicited comments require ten distinct moments a player wants to point at — one directive can't produce that variance.

Character silence is honored structurally (no audio, Silas speaks in text, reduced-motion respected) but is not yet *load-bearing*. The silence-as-horror pillar (§4) requires the player to feel Silas withdraw — which requires directives whose consequences would make Silas withdraw, which requires Track C content.

**Honest read:** V5 was chrome work. It brought the HUD to mockup parity and gave the trace ledger its bottom-edge voice. It did not — and could not — earn the ≥10-signal threshold because that threshold is a *content* threshold, not a *chrome* threshold. This is exactly the case `docs/ship-gate.md:49` describes: "If NO, do not ship — return to `design-discovery` skill and identify what's missing."

What's missing is Track C (Q1 content fill) + Track B5 (real module effects) + Track D (comfort at runtime). Those three tracks, together, are what turns the current chrome-complete slice into a §17-passing slice.

Verdict: **DO NOT SHIP.** Return to plan; execute Track C next.
